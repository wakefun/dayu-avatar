## Findings

| ID | Severity | Type | Evidence | Why it matters | Suggested non-breaking fix | Destructive / risky fix |
|---|---|---|---|---|---|---|
| F-01 | High | Security / privacy contract | Frontend treats media URLs as displayable public strings: `apps/web/src/lib/types.ts:23`, `66`, `111`; components render them directly at `UploadCard.tsx:65`, `Cards.tsx:98`, `ResultPage.tsx:48/75/140`. Backend serves all uploaded/generated media publicly with `express.static` at `apps/api/src/index.ts:325-326`. | Uploaded personal reference photos and generated avatars are session-owned in API routes, but their `/static/...` URLs are bearerless. Anyone with the URL can fetch media outside the authenticated API contract. | Interim low-risk mitigation: set explicit `Referrer-Policy` and private/no-store cache headers for static media, avoid leaking media URLs into third-party requests, and document that media URLs are not secret. | Real privacy fix is a contract change: replace public `/static` media with authenticated media endpoints or short-lived signed URLs. This affects `Asset.fileUrl`, `GalleryItem.imageUrl`, `thumbnailUrl`, `User.avatarUrl`, image rendering, downloads, caching, and service worker behavior. |
| F-02 | Medium | Security hardening | Grep found no `Content-Security-Policy`, `Referrer-Policy`, `Permissions-Policy`, `X-Frame-Options`, `helmet`, or equivalent frontend delivery headers. `apps/web/index.html` contains metadata but no CSP. | React escaping lowers XSS risk, but no browser-enforced policy limits script/image/font/connect origins if an injection bug appears later. Referrer policy also matters because media URLs are currently public. | Add low-risk headers first: `Referrer-Policy: no-referrer` or `strict-origin-when-cross-origin`, `X-Content-Type-Options: nosniff`, conservative `Permissions-Policy`, and optionally report-only CSP. | Strict enforcing CSP can break Vite assets, Google Fonts, inline styles generated by tooling, image previews, or future OIDC/provider redirects if not tested carefully. |
| F-03 | Medium | Privacy / performance / PWA offline | `apps/web/src/styles.css:1` imports Google Fonts via `@import url('https://fonts.googleapis.com/...')`; the service worker only handles same-origin requests (`apps/web/public/sw.js:20`). | Each app load can contact Google font endpoints, leak network metadata, fail or slow down in restricted networks, and is not covered by same-origin PWA caching. | Prefer bundled/self-hosted fonts or system font stack fallback. If retaining external fonts, add explicit CSP/font-src/connect-src allowances and preload strategy. | Removing/replacing fonts changes visual output; self-hosting adds binary assets and license/caching considerations. |
| F-04 | Medium | Accessibility bug | `apps/web/src/components/UploadCard.tsx:51` and `85` render file inputs as `className="hidden"` inside a clickable `<label>`. | Pointer users can click the label, but keyboard users cannot tab to a `display:none` input or focusable upload control, making the primary upload path inaccessible. | Use a visually-hidden but focusable file input, or make the label a keyboard-operable button-like control with `tabIndex`, `role="button"`, and Enter/Space handling while preserving click-to-upload. | Reworking upload markup broadly can alter the 1/2/3-image layout contract; keep the square-card behavior unchanged. |
| F-05 | Medium | Accessibility bug | Closed side drawer is only transformed offscreen and `aria-hidden`: `apps/web/src/components/AppShell.tsx:76-82`; it still contains focusable `NavLink`s and the install button at `99-117`. | Keyboard/screen-reader users may tab into hidden/offscreen navigation controls when the drawer is closed. | When closed, render drawer conditionally, add `inert`, or set tab stops disabled for drawer descendants; keep overlay close behavior when open. | A large navigation refactor or route restructuring is unnecessary and riskier. |
| F-06 | Medium | Accessibility bug | `ImageLightbox` declares `role="dialog"` and `aria-modal="true"` at `apps/web/src/components/ImageLightbox.tsx:31-32`, closes on backdrop at `33`, but has no accessible dialog label, focus trap/initial focus, Escape handler, or focus restore. | Modal preview/actions can be hard to operate with keyboard or assistive tech, especially because destructive/gallery actions live in the dialog footer. | Add `aria-label`/`aria-labelledby`, set initial focus to close/dialog, handle Escape, and restore prior focus on close. | Full focus-trap/inert background implementation needs browser testing; avoid changing action semantics at the same time. |
| F-07 | Medium | UX bug / async error handling | API actions often await without local `try/catch` or pending guards: `GalleryPage.tsx:30`, `73`, `82`; `QueuePage.tsx:14-15`, `36`; `HistoryPage.tsx:17-18`; `ResultPage.tsx:64`; `SettingsPage.tsx:44`. | Network/API failures can become unhandled promise rejections or silent no-ops, leaving stale UI after favorite/delete/save/logout/retry actions. | Add local error state and disable/re-enable pending action buttons around each mutation/load; keep existing API contracts. | Introducing a global data-fetching library or broad state refactor would be larger than needed. |
| F-08 | Medium | Result route bug | `ResultPage` calls `api.getTask`, `api.getTaskResult`, and `api.getGallery` in a single `Promise.all` at `apps/web/src/pages/ResultPage.tsx:21-24`; `/result` returns 409 unless task is completed (`apps/api/src/index.ts:651-664`). The page still renders copy saying “结果已准备完成” at `ResultPage.tsx:38`. | Opening `/generate/result/:taskId` directly for queued/processing/failed/missing tasks rejects the whole load, loses task status, and shows result-ready framing plus an error rather than redirecting to progress or terminal state. | Fetch task first; if status is queued/processing navigate to loading, if failed/canceled show terminal state or queue CTA, and only fetch result/gallery after completed. | Changing route names or backend result semantics is unnecessary. |
| F-09 | Low | Polling quality / optimization | `QueuePage` polls every 2s (`apps/web/src/pages/QueuePage.tsx:18-22`) and `load` has no in-flight guard or error catch (`12-15`). | Slow responses can overlap; an older response may overwrite newer queue state, and transient failures are unhandled. Polling also continues while the tab is hidden. | Track in-flight request or ignore stale sequence numbers, catch errors, and optionally pause polling when `document.visibilityState === 'hidden'`. | Replacing with SSE/WebSockets or a data-fetching framework is a larger behavior change. |
| F-10 | Low | API wrapper quality / CORS optimization | `request<T>` always sends `Content-Type: application/json` including GETs (`apps/web/src/lib/api.ts:14-18`), then spreads `...init` after the default headers at `20`, so future `init.headers` would replace the merged header object. | Unnecessary JSON content type on GET can trigger avoidable CORS preflights in cross-origin deployments, and the header merge pattern is misleading/fragile for future callers. | Destructure `headers` out of `init`, spread `...rest` first, and set `Content-Type` only when sending a JSON body and not when using `FormData`. | Changing all API helper signatures is unnecessary. |
| F-11 | Low | Contract drift / regenerate prefill | `GeneratePrefillState` only preserves `generationParams.size` (`apps/web/src/pages/GeneratePage.tsx:77-85`, `120`), while shared types include `model`, `quality`, `size`, `outputFormat` (`apps/web/src/lib/types.ts:39-43`, `96-100`), and task creation always resubmits hard-coded `model: 'gpt-image-2'`, `quality: 'high'`, `outputFormat: 'png'` (`GeneratePage.tsx:220-224`). | History/result “重新生成” claims to preserve generation settings in specs, but hidden params can drift if backend defaults or historical tasks differ. | Store prefilled generation params in local state, even if only size has visible controls, and submit preserved model/quality/outputFormat unless the user changes visible settings. | Adding full model/quality/output-format UI controls or changing backend defaults is broader product behavior. |
| F-12 | Medium | Shared contract quality | Frontend contracts are manually duplicated in `apps/web/src/lib/types.ts`; backend maps the same shapes separately in `apps/api/src/index.ts:2020-2180`. Specs say `pnpm typecheck` should fail if backend/frontend task contracts drift (`.trellis/spec/frontend/type-safety.md:72-77`), but no shared contract package or contract tests were found. | TypeScript validates each app independently, so backend response drift can compile cleanly and fail only at runtime. | Add lightweight contract tests or schema snapshots comparing representative API responses to frontend expectations; alternatively introduce a small shared contracts module when scope allows. | Moving to generated clients or a new shared package touches workspace structure and import boundaries; treat as planned work. |
| F-13 | Low | UI copy / auth contract mismatch | `SessionSummary.authMode` can be `'mock' | 'oidc'` (`apps/web/src/lib/types.ts:1-6`), backend returns `authMode: 'mock'` for mock login (`apps/api/src/index.ts:446-447`), but settings fallback label says `当前会话由大宇统一登录接入` for non-OIDC at `apps/web/src/pages/SettingsPage.tsx:18`. | In mock/dev mode, account settings presents the session as unified-login-connected, which can mislead testers/users about auth state. | Use explicit labels for `oidc`, `mock`, and missing session, e.g. “模拟登录/体验模式”. | Renaming auth modes or changing backend auth behavior is unnecessary. |
| F-14 | Low | PWA reliability / developer UX | Service worker registers unconditionally when supported (`apps/web/src/main.tsx:14-17`); registration promise is ignored without catch. `sw.js` caches shell files during install (`apps/web/public/sw.js:1-5`). | In development or failed registration/install cases, stale caches or unhandled registration failures can make debugging and refresh behavior confusing. | Register only in production or catch/log registration failures; consider cache version discipline for shell changes. | Removing PWA support or changing offline strategy broadly is higher-risk. |
| F-15 | Medium | Test coverage gap | No frontend test files or test tooling references were found; specs require smoke coverage for login, drawer navigation, generation controls, upload, result, gallery, queue/history/settings (`.trellis/spec/frontend/component-guidelines.md:107-113`, `.trellis/spec/frontend/directory-structure.md:54-57`). | Current lint/typecheck/build pass, but UI flows and cross-layer contracts have no automated regression coverage. | Add a minimal smoke test plan or first automated tests around route rendering and critical action buttons. | Adding new test dependencies/runner may affect lockfile and CI expectations; coordinate if dependency changes are considered risky. |
| F-16 | Low | Low-risk performance optimization | `App` refreshes session with `api.me()` inside an effect keyed by `location.pathname` (`apps/web/src/App.tsx:38`, `44-60`), so every route change makes an auth request. | Extra requests add latency/load and can create noisy logs; auth state is usually stable between explicit login/logout/avatar updates. | Refresh once on mount and after explicit auth-affecting actions, or cache session until failure/visibility changes. | Introducing global auth state machinery is unnecessary for this app size. |
| F-17 | Low | Upload UX optimization | `UploadCard` only uses HTML `accept="image/png,image/jpeg,image/webp"` (`apps/web/src/components/UploadCard.tsx:4`, `51`, `85`); backend enforces file type and 10MB limit (`apps/api/src/index.ts:297-300`, `510-514`). | Users can still select/attempt invalid or oversized files depending on platform; errors only arrive after upload. | Add client-side file size/type checks before `api.upload` and show the existing error message area. Keep backend validation authoritative. | Changing backend file-size limits or accepted formats affects API behavior and should be coordinated. |
